## Prioritization rationale

1. **#380 (hydration mismatch)** — ships a console warning on every auth-protected page load.
   High visibility for anyone evaluating the framework. Target: early v1.1.
2. **#326 (middleware ordering)** — architectural technical debt. Attribute-based discovery
   exists but the manual fallback is fragile. Fix before adding new middleware.
3. **#339 (hardcoded version)** — trivial; wire VERSION file to framework output.
4. **#342 (API discovery)** — useful for SDK consumers and Telescope.
5. **#335/#336 (admin polish)** — low risk, improves day-to-day UX.
6. **#337 (GraphQL stub)** — may stay a stub until v1.2 unless a contributor picks it up.

## Notes on #380 — hydration mismatch

**Symptom:** Vue emits a hydration mismatch warning on auth-protected pages when the global
middleware runs differently between SSR and client.

**Root cause hypothesis:** `checkAuth()` result is available synchronously on the server but
arrives asynchronously on the client, causing the initial render tree to diverge.

**Candidate fix:** Use `useState()` to persist the auth result across the SSR→client boundary,
or delay the client-side redirect until `onMounted` to avoid the mismatch entirely.

**Test approach (TDD):**
1. Write a Playwright test that captures console warnings on page load for `/`
2. Confirm the test fails (warning present)
3. Apply fix
4. Confirm test passes (no warning)
