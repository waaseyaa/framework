## Issue #380 — Vue hydration mismatch on auth-protected pages

**Symptom:** Vue emits a hydration mismatch warning when the global auth middleware
runs differently between SSR and client on auth-protected pages.

### Branch
`fix/380-hydration-mismatch`

### TDD Steps

1. **Write failing Playwright test** (`packages/admin/e2e/auth-hydration.spec.ts`):
   ```ts
   test('no hydration mismatch on protected page', async ({ page }) => {
     const warnings: string[] = []
     page.on('console', msg => {
       if (msg.type() === 'warning' && msg.text().includes('Hydration')) {
         warnings.push(msg.text())
       }
     })
     await mockUserMeRoute(page)
     await page.goto('/')
     await page.waitForLoadState('networkidle')
     expect(warnings).toHaveLength(0)
   })
   ```
2. **Run** — confirm test fails (warning present)
3. **Implement fix** — candidate approaches:
   - Use `useState('auth-checked', () => false)` to persist auth state across SSR→client
   - Move redirect logic entirely to `onMounted` to avoid SSR/client tree divergence
   - See `packages/admin/app/middleware/auth.global.ts` and `packages/admin/app/composables/useAuth.ts`
4. **Run** — confirm test passes (no warning)

### Expected file changes
- `packages/admin/app/middleware/auth.global.ts` — defer redirect to client-only
- `packages/admin/app/composables/useAuth.ts` — persist auth state with `useState()`
- `packages/admin/e2e/auth-hydration.spec.ts` — new test file

### SSR/SPA impact
- SSR: no server-side auth fetch removed — existing guard preserved
- Client: `onMounted` redirect fires correctly after hydration completes

### Verification criteria
- [ ] New Playwright test passes
- [ ] No Vue hydration warning in browser console on `/`
- [ ] Existing auth e2e tests still pass
- [ ] `npm test` (78 tests) still green

## Issue #326 — Middleware uses manual ordering instead of attribute-based discovery

**Symptom:** `#[AsMiddleware(priority: N)]` attribute exists but some middleware
ordering still relies on manual registration order.

### Branch
`fix/326-middleware-ordering`

### TDD Steps

1. **Write failing unit test** (`packages/foundation/tests/Unit/Middleware/MiddlewareOrderingTest.php`):
   ```php
   #[Test]
   public function middleware_order_is_determined_solely_by_attribute_priority(): void
   {
       // Register middleware in reverse-priority order
       // Assert resolved pipeline order matches attribute priorities, not registration order
   }
   ```
2. **Run** — confirm test fails
3. **Trace** `PackageManifestCompiler` middleware discovery → `AbstractKernel` pipeline assembly
4. **Fix** — ensure `#[AsMiddleware(priority:)]` is the sole ordering mechanism, no manual fallback
5. **Run** — confirm test passes

### Expected file changes
- `packages/foundation/src/Kernel/AbstractKernel.php` — remove manual ordering fallback
- `packages/foundation/src/Middleware/AsMiddleware.php` — verify attribute is required (no default priority)
- `packages/foundation/tests/Unit/Middleware/MiddlewareOrderingTest.php` — new test

### Backend impact
- All existing middleware must have explicit `priority:` values in `#[AsMiddleware]`
- Audit: `SessionMiddleware`, `AuthorizationMiddleware`, `CorsMiddleware`, `CsrfMiddleware`

### Verification criteria
- [ ] New unit test passes
- [ ] All existing middleware have explicit `priority:` values
- [ ] PHP test suite (4352 tests) still green
