Date: 2026-03-04  
Scope: `#130 v1.1: Query-path profiling + hot-path optimization (API/SSR/discovery)`

## Profiling Focus

Dominant read-path focus areas reviewed for v1.1:

- discovery/SSR graph browse composition (`RelationshipTraversalService::browse`)
- MCP read-path calls (already optimized/cached in `#129`)
- public API discovery endpoints using relationship traversal

## Hot-Path Finding

In `RelationshipTraversalService::browse`, repeated edges that reference the same related entity triggered repeated `load()` calls for that endpoint during edge mapping.

Impact surfaces:

- discovery API payload assembly (`hub/cluster/timeline/endpoint`)
- SSR relationship navigation context composition
- MCP discovery/traversal consumers that rely on relationship browse output

## Optimization Delivered

- Added in-request related-entity summary cache in `RelationshipTraversalService`:
  - key: `{entity_type}:{entity_id}`
  - reused across outbound + inbound browse mapping in the same request path
- No contract changes to traversal/discovery/SSR payloads.
- Deterministic ordering and visibility semantics preserved.

## Auditable Evidence

Added deterministic regression/profiling test:

- `RelationshipTraversalServiceTest::testBrowseCachesRepeatedRelatedEntitySummaryLoadsAcrossDirections`
  - scenario: 3 edges referencing the same related endpoint across outbound/inbound sets
  - assertion: related entity storage `load()` calls collapse to `1`
  - equivalent pre-optimization behavior would perform one load per edge (`3`)

Observed load-call reduction in this scenario:

- before: `3` endpoint loads
- after: `1` endpoint load
- reduction: `66.7%`
